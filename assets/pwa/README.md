# PWA foundation files

These files make mobile-cc installable as a PWA on Android (and on
desktop via Chrome's install prompt). **Since 2026-06-10 they are
served by the daemon**: `main.rs` embeds them via the `PWA_ASSETS`
table and passes them through `ttyview-core`'s
`RunOptions.extra_static` hook (which landed upstream that day â€”
the "How it wired up" section below records the original design).
`GET /api/instance` advertises `manifest_url` + `service_worker_url`;
the web client injects the manifest link and registers the SW.

## What's here

| File | Purpose |
|---|---|
| `manifest.webmanifest` | PWA manifest â€” name, theme color (Terminal Green `#00ff66`), background `#0a0f0a`, icon set. `display: standalone`, scope = `/`. |
| `sw.js` | Minimal service worker â€” install + activate + fetch-passthrough. No caching at this stage (would silently serve stale CC transcript / pane state). Skeleton for Web Push handlers documented inline. |
| `icons/icon-{192,512}.png` | "Any" purpose icons. Centered `mcc_` wordmark on near-black background. |
| `icons/icon-{192,512}-maskable.png` | Maskable variants with the 80% safe zone. Android home-screen masks crop the outer 20%; main content sits well inside. |

The icons are bytewise stable (generated by `assets/pwa/icons/gen.py`
which is *not* committed â€” they were one-shot PIL outputs). If you
want to change the design, regenerate with PIL or replace with hand-
authored assets at the same sizes / paths.

## How it wired up (originally "the upstream blocker" â€” landed 2026-06-10)

> Resolution: ttyview-core grew `RunOptions.extra_static:
> Vec<(String, Vec<u8>)>` (simpler than the trait proposed below â€”
> assets are baked bytes, no dynamic provider needed), served by the
> router's fallback handler. Items 2 and 3 landed exactly as
> proposed, gated on `/api/instance`'s `manifest_url` /
> `service_worker_url` fields. Original design kept for context:

`ttyview-core` serves its bundled UI assets via `rust-embed`. mobile-cc
currently has zero way to ship additional static assets through that
pipeline â€” the embedder API today only takes plugin sources and a
plugin manifest, not arbitrary top-level files.

What's needed upstream:

1. **`ttyview-core` `RunOptions.extra_static_assets: Option<Arc<dyn StaticAssetProvider>>`**
   (or similar) â€” a hook that lets embedders inject a directory of
   files to serve at a given URL prefix (proposal: `/pwa/*` and
   `/manifest.webmanifest` + `/sw.js` at root). Same shape as the
   plugin bundle: a trait with `get(path) -> Option<Cow<[u8]>>`.

2. **`<link rel="manifest">`** added to `ttyview-core/ui/index.html`,
   pointing at `/manifest.webmanifest`. Conditional on the daemon
   exposing a manifest (use a small `/api/instance` field like
   `manifest_url`).

3. **Service worker registration** added to the boot loader in
   `ui/index.html`. `navigator.serviceWorker.register('/sw.js')`,
   guarded by `navigator.serviceWorker` existing.

Once those land, mobile-cc's binary embeds this `assets/pwa/`
directory via `rust-embed` and hands it to ttyview-core through
`extra_static_assets`. The current binary doesn't need any wiring
beyond a one-line `pub use` and the embedded bundle.

Tracking notes for the upstream PR: this is in
`mobile-cc/CLAUDE.md` under **Planned: PWA + Web Push** and
`ttyview/.claude/mobile-cc-binary.md` (architecture obligations).

## What's *not* here yet

- **Web Push** (VAPID keypair generation, daemon-side subscription
  store, `/api/push/*` endpoints, `push` + `notificationclick` SW
  handlers). The headline PWA feature for our use case â€” "CC asked
  for permission, your phone buzzes" â€” depends on this. Foundation-
  only for now; planned scope detail in `mobile-cc/CLAUDE.md`.
- **Offline-shell caching.** Serving stale plugin JS / index.html
  after a daemon upgrade has subtle correctness traps (mismatched
  plugin schema between cached JS and live `/api/instance`). Punt
  until there's a real demand.
- **Deep-link routing.** `?pane=%N` is documented in the PWA scope
  but lives in the page UI, not the SW.
